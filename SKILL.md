# Codebase Generator — 大型项目文档树生成器

为大型代码库生成分层结构化文档树。输出不是单个文件，而是一个结构化目录，
包含各级别的索引文件（toc.md）和spec文档。

**注意**：最后生成的所有文档必须是中文！

## 输入参数

用户需提供以下两个参数：

1. **项目路径**：待分析的源代码根目录（即大型项目的根路径）。
2. **文档树路径**：生成的文档树输出目录。

如果用户未明确指定，询问用户确认。

## 文档集合概念

### 索引文件（toc.md）

索引文件是各级别的导航文件，控制其内容相对精简，**不超过500行**，因为它本质上是一个索引。
所有层级（总纲、模块、子模块）都有各自的索引文件。

### 总纲

- **路径**：`{文档树路径}/toc.md`
- **约束**：索引文件，不超过500行
- **内容**：
  - 项目详细介绍（项目背景、业务目标、整体设计思路）
  - 对项目涉及的所有`模块`的介绍，包括：
  - 每个模块的详细简介（说明该模块的职责、覆盖的业务领域、关键功能）
  - 每个模块涉及的`子模块`的详细简介（名称+该子模块的职责说明、覆盖范围）
  - 每个模块涉及的`spec`文档的详细简介（名称+该spec覆盖的功能/接口说明）
  - 每个`模块`索引文件的路径链接
  - **重要**：总纲是检索器（retriever）查询的入口，描述必须足够详细以支持顺滑的导航路径。例如：用户搜索"mcp的实现"，总纲中的模块/子模块/spec描述应能准确指向正确的下一级索引，确保查询路径为 总纲 → 模块索引 → 子模块索引 → spec文档。

### 模块

- **定义**：一个比较大的领域聚集，比如工具模块、插件模块、hook模块
- **划分原则**：模块划分应综合考虑**架构层面**（代码组织、职责划分）、**业务层面**（领域逻辑、功能边界）、**目录结构**（源码目录分布）以及**整体视角**（模块之间的关系、依赖）。不要过于依赖目录结构，目录只是参考之一，关键是理解代码的实际业务和架构内涵。
- **约束**：模块下面可以有很多`子模块`文件夹或者`spec`文档，但`子模块`或`spec`文档总数不超过30
- **路径**：`{文档树路径}/{模块名}/`，比如工具模块 → `{文档树路径}/tools/`

### 模块索引文件

- **路径**：`{文档树路径}/{模块}/toc.md`
- **约束**：索引文件，不超过500行
- **内容**：
  - 该模块的**模块关系图**（ASCII 图，展示本模块内子模块之间的调用/依赖关系）
  - 对该模块涉及的所有`子模块`/`spec`文档的详细简介（职责说明、覆盖范围），以及`子模块`索引文件路径（如 `./子模块/toc.md`）/`spec`文档路径
  - 描述必须足够详细以支持检索器导航

### 子模块

- **定义**：比较小的领域聚集
- **划分原则**：子模块划分同样应综合考虑**架构层面**、**业务层面**、**目录结构**以及**整体视角**。不要过于依赖目录结构，应根据代码的实际业务逻辑和架构内聚性来判断。
- **约束**：下面包含 5~40 个`spec`文档（注意，只有spec文档，不再嵌套子模块）
- **路径**：`{文档树路径}/{模块}/{子模块}/`

### 子模块索引文件

- **路径**：`{文档树路径}/{模块}/{子模块}/toc.md`
- **约束**：索引文件，不超过500行
- **内容**：
  - 该子模块的**spec 关系图**（ASCII 图，展示本子模块内各 spec 之间的调用/依赖关系）
  - 对该子模块涉及的所有`spec`文档的详细简介（覆盖的功能/接口说明）和`spec`文档路径
  - 描述必须足够详细以支持检索器导航

### spec 文档

- **定义**：最小的领域粒度。对于同一最小业务逻辑、相同逻辑或相关性强的一个或数个源代码文件的spec详情，涉及到的数个`功能/接口`集合。**必须详实的介绍这些功能和接口spec**
- **示例**：`spec-read.md` 是 read 工具的spec文档
- **内容要求**：
  - 功能详情/接口详情
  - 该功能/接口核心代码
  - 该功能/接口代码范围：`{源码路径}:起始行号-结束行号`

## 文档树结构

```
{文档树路径}/
├── toc.md                          ← 总纲（总索引，≤500行）
├── {模块-a}/
│   ├── toc.md                      ← 模块索引（模块级索引，≤500行）
│   ├── {子模块-1}/
│   │   ├── toc.md                  ← 子模块索引（子模块级索引，≤500行）
│   │   ├── spec-xxx.md
│   │   └── spec-yyy.md
│   ├── {子模块-2}/
│   │   ├── toc.md
│   │   └── ...
│   └── spec-zzz.md                 ← 直接挂在模块下的spec
├── {模块-b}/
│   ├── toc.md
│   └── ...
└── ...
```

## 参考资料

### Spec 文档生成规范

> **！！！重要！！！每次生成 spec 文档之前，必须重新读取以下文件，严格遵循其规范。**
>
> **`references/SPEC-GENERATION-GUIDE.md`** — spec 文档生成的标准规范，包含核心原则、源文件阅读范围、必须包含的内容章节、完整格式模板。
>
> **不得跳过此步骤**。即使之前已读过，在处理后续 spec 文档时，仍需重新读取该文件，以确保始终遵循规范。

## 执行阶段

严格按以下阶段执行。不得跳过或合并阶段。

### 阶段一：全局扫描 — 发现所有模块（用临时中间文件记录）

1. 读取项目顶层结构：目录列表、关键配置文件（package.json、Cargo.toml、go.mod等）、README。
2. 识别项目的主要领域区域。**划分模块时，不要过于依赖目录结构**，应综合考虑架构层面（代码组织、职责划分）、业务层面（领域逻辑、功能边界）、目录结构以及整体视角（模块间的关系和依赖）。每个领域区域对应一个**模块**。
3. 用临时中间文件记录采集的信息。创建 `{文档树路径}/.tmp/module-plan.json`：
   ```json
   {
     "modules": [
       {
         "name": "模块文件夹名称",
         "display_name": "可读名称",
         "description": "详细描述该模块的职责、覆盖的业务领域、关键功能",
         "source_paths": ["src/module-folder/"],
         "estimated_children": "子模块数 + spec数"
       }
     ]
   }
   ```
4. 向用户展示模块规划，等待确认后再继续。

### 阶段二：逐模块扫描 — 发现子模块和spec（用临时中间文件记录）

对每个模块**依次处理**（一次只处理一个模块）：

1. 深入扫描模块的源码路径。读取关键文件，理解领域。
2. 判断哪些子项应作为**子模块**（5–40个相关spec的集合），哪些作为独立的**spec文档**（直接挂在模块下）。划分时应综合考虑架构层面、业务层面、目录结构及整体视角，不要过于依赖目录结构。
3. 创建/更新中间文件 `{文档树路径}/.tmp/module-{name}-plan.json`：
   ```json
   {
     "module": "模块名称",
     "children": [
       {
         "type": "submodule",
         "name": "子模块文件夹名称",
         "display_name": "可读名称",
         "description": "详细描述",
         "source_paths": ["src/module-folder/subfolder/"],
         "specs": [
           {
             "name": "spec-文件名.md",
             "display_name": "功能名称",
             "description": "详细描述覆盖的功能/接口说明",
             "source_files": ["src/module-folder/subfolder/file.ts"],
             "line_range_hint": "大致起始-结束行"
           }
         ]
       },
       {
         "type": "spec",
         "name": "spec-功能.md",
         "display_name": "功能名称",
         "description": "详细描述覆盖的功能/接口说明",
         "source_files": ["src/module-folder/standalone-file.ts"],
         "line_range_hint": "大致起始-结束行"
       }
     ]
   }
   ```

### 阶段三：逐子模块扫描 — 验证并补充阶段二规划的子模块和直接spec（用临时中间文件记录）

对每个子模块**依次处理**：

1. 读取阶段二生成的中间文件（module-{name}-plan.json）中该子模块及其 spec 的规划信息。
2. 深入扫描子模块的源码路径，验证规划中的子模块名、spec 名称、描述、源文件路径等信息是否准确，如有遗漏则补充。

### 阶段四：正式处理 — 先生成总纲文档

在进入模块处理之前，先根据阶段一采集的模块信息，生成初始版本的 `{文档树路径}/toc.md`：

```markdown
# {项目名称} — 文档树

> 项目： `{项目路径}` 的项目文档。

｛项目简要描述｝


## 模块总览

### {模块A名称}

> {模块A详细简介：职责、覆盖的业务领域、关键功能}

涉及子模块/spec：（待补充）

索引：[模块A/toc.md](./module-a/toc.md)

### {模块B名称}

> {模块B详细简介：职责、覆盖的业务领域、关键功能}

涉及子模块/spec：（待补充）

索引：[模块B/toc.md](./module-b/toc.md)
```

此版本为初始骨架版本，子模块/spec 详情待阶段五处理完后再补充完整（见阶段五-5.6）。
注意：总纲必须列出每个模块的详细简介和模块索引文件路径。

### 阶段五：按照模块顺序，严格依次处理

按照模块顺序，**严格依次处理**，先处理完一个模块后才能处理下一个模块。

#### 5.1 处理模块下的子项

处理模块时，按照模块下面的`子模块`、`spec`文档顺序依次处理。
处理完一个子模块后才能处理下一个子模块或者spec。

#### 5.2 处理子模块下的spec

处理子模块时，需要按照这个子模块下的`spec`文档顺序依次处理。
处理完一个spec后才能处理下一个spec。

#### 5.3 生成 spec 文档

> **！！！警告！！！** 每次生成任何 spec 文档之前，必须重新读取 `references/SPEC-GENERATION-GUIDE.md`。
> 不得依赖记忆或之前的上下文。每次都要重新读取，严格遵循其中所有规范。
> 本步骤不可跳过。

**如使用 subagent 处理 spec 任务**：必须在 subagent 的 prompt 中明确指示它在做任何 spec 工作之前，先读取并严格遵循 `references/SPEC-GENERATION-GUIDE.md`。subagent 没有主会话记忆，必须通过 prompt 强制约束。

详细的生成规范（核心原则、源文件阅读范围、内容章节、格式模板）见 `references/SPEC-GENERATION-GUIDE.md`。

每写完一个spec，输出进度信息：`[spec] 模块/子模块/spec-xxx.md — 完成`

#### 5.4 处理完子模块下面的子项后，创建子模块索引文件

子模块下所有spec文档写完后，创建 `{文档树路径}/{模块}/{子模块}/toc.md`：

```markdown
# {子模块可读名称}

> {子模块详细简介：职责说明、覆盖范围}

## Spec 关系图

```
┌─────────────────────────┐
│       {Spec A 名称}              │
│       {覆盖的功能简述}            │
└───────────┬─────────────┘
            │
            ▼
┌─────────────────────────┐
│       {Spec B 名称}              │
│       {覆盖的功能简述}            │
└─────────────────────────┘
```

## Spec 文档

| Spec | 描述 | 源文件 |
|------|------|--------|
| [spec-xxx.md](./spec-xxx.md) | {详细描述：覆盖的功能/接口说明} | `src/.../file.ts` |
| [spec-yyy.md](./spec-yyy.md) | {详细描述：覆盖的功能/接口说明} | `src/.../file2.ts` |
```

#### 5.5 处理完模块下面的子项后，创建模块索引文件

模块下所有子项（子模块和直接spec）处理完后，创建 `{文档树路径}/{模块}/toc.md`：

```markdown
# {模块可读名称}

> {模块详细简介：职责、覆盖的业务领域、关键功能}

## 模块关系图

```
┌─────────────────────────┐
│     {子模块A / 直接spec名称}      │
│     {描述或职责简述}              │
└───────────┬─────────────┘
            │
            ▼
┌─────────────────────────┐
│     {子模块B / 直接spec名称}      │
│     {描述或职责简述}              │
└─────────────────────────┘
```

## 子模块

| 子模块 | 描述 | Spec数 | 索引 |
|--------|------|--------|------|
| [submodule-1/](./submodule-1/) | {详细描述：职责说明、覆盖范围} | N个spec | [toc.md](./submodule-1/toc.md) |
| [submodule-2/](./submodule-2/) | {详细描述：职责说明、覆盖范围} | M个spec | [toc.md](./submodule-2/toc.md) |

## 直接 Spec 文档

| Spec | 描述 | 源文件 |
|------|------|--------|
| [spec-zzz.md](./spec-zzz.md) | {详细描述：覆盖的功能/接口说明} | `src/.../file.ts` |
```

#### 5.6 所有模块处理完后，再次更新总纲文档

所有模块都处理完成后，重新写入 `{文档树路径}/toc.md`，这次要把完整的模块详细简介、
每个子模块的详细简介、每个spec的详细简介、模块索引路径全部写入，替换掉阶段四的初始版本：

```markdown
# {项目名称} — 文档树

> 自动为 `{项目路径}` 生成的项目文档。

## 模块总览

### {模块A名称}

> {模块A详细简介：职责、覆盖的业务领域、关键功能}

- **子模块**
  - `{子模块1}` — {子模块1详细简介：职责说明、覆盖范围}（N个spec）
  - `{子模块2}` — {子模块2详细简介：职责说明、覆盖范围}（M个spec）
- **直接spec**
  - `spec-xxx.md` — {spec详细简介：覆盖的功能/接口说明}
  - `spec-yyy.md` — {spec详细简介：覆盖的功能/接口说明}
- **索引文件**：[模块A/toc.md](./module-a/toc.md)

### {模块B名称}

> {模块B详细简介：职责、覆盖的业务领域、关键功能}

- **子模块**
  - `{子模块3}` — {子模块3详细简介：职责说明、覆盖范围}（K个spec）
- **直接spec**
  - `spec-zzz.md` — {spec详细简介：覆盖的功能/接口说明}
- **索引文件**：[模块B/toc.md](./module-b/toc.md)
```

注意：总纲中每个模块必须包含其子模块的详细简介和直接spec的详细简介，不能只写数量。简介应说明职责和覆盖范围，而非仅名称。描述必须足够详细以支持检索器导航——用户给出一个查询（如"mcp的实现"），应能通过总纲 → 模块索引 → 子模块索引 → spec文档 的路径顺利定位。

#### 5.7 清理临时中间文件

所有文档生成完成后，删除 `{文档树路径}/.tmp/` 目录及其下所有中间规划文件。

#### 5.8 全面梳理 — 完善总纲文档

所有 spec 和索引文档生成完成后，应对整个项目文档进行全面梳理和优化。

**核心任务**：

1. **内容审查**：仔细阅读所有生成的 spec 文档，理解各模块、子模块之间的联系与依赖关系。
2. **架构梳理**：基于对全部 spec 的理解，重新梳理项目架构，包括：
   - 项目的核心入口点
   - 主要模块之间的调用关系
   - 关键模块在整体架构中的角色
   - 梳理出项目的**模块依赖关系图**（分层 ASCII 图），清晰展示各层的职责和层间依赖，例如：

     ```
     ┌─────────────────────────────────────┐
     │           第一层：入口 (Entry)        │
     │         main.tsx / CLI 入口          │
     └──────────────────┬──────────────────┘
                        │
                        ▼
     ┌─────────────────────────────────────┐
     │      第二层：基础设施 (Foundation)    │
     │    config / state / migrations      │
     └──────────────────┬──────────────────┘
                        │
                        ▼
     ┌─────────────────────────────────────┐
     │      第三层：输入输出 (I/O)           │
     │        ui / cli / bridge            │
     └──────────────────┬──────────────────┘
                        ▼
     ...
     ```

3. **总纲优化**：更新 `{文档树路径}/toc.md`，补充：
   - 项目详细介绍（项目背景、业务目标、整体设计思路）
   - **模块依赖关系图**（分层 ASCII 图，展示各层职责和层间依赖）
   - 入口点说明
   - **尽量精简但不失信息量**：500行是建议上限，但总纲承担着检索器（retriever）导航的角色，信息完整性优先于行数控制。如需在500行内取舍，应优先保证模块关系、入口说明、调用链等关键导航信息完整。
4. **划分调整（如需要）**：在全面理解项目后，允许对已有 spec 的内容、边界进行修订；
   如发现模块/子模块划分不合理，也可适当调整，但调整前需向用户说明原因。

> 注意：此阶段的价值在于——在逐个生成 spec 的过程中，我们对项目的理解是局部的；
> 现在所有 spec 已生成，站在全局视角重新审视，往往能发现更准确的架构描述和模块划分方式。

#### 5.9 最终检查 — 验证索引可追溯性

在完成所有文档生成和全面梳理后，必须对所有 toc.md 进行索引可追溯性验证，确保从任意一级索引出发，都能顺利找到最终的 spec 和源码。

**验证内容**：

1. **toc.md 检查**：从总纲开始，随机选取 3–5 个模块描述，验证能否通过描述准确找到对应的模块 toc → 子模块 toc → spec
2. **toc.md 检查**：从模块索引随机选取 2–3 个子模块/spec 描述，验证描述是否准确指向正确的 spec
3. **toc.md 检查**：从子模块索引随机选取 2–3 个 spec 描述，验证能否定位到正确的源文件
4. **spec 文档检查 — 模拟检索流程**：从实际使用角度出发，给出若干个功能的部分描述（如"mcp的实现"、"文件读取"、"状态管理"等），验证能否从总纲出发逐级导航找到对应的 spec 和源码：
   - 部分描述 → 总纲 → 模块索引 → 子模块索引 → spec 文档
   - 找到 spec 后检查是否符合 SPEC-GENERATION-GUIDE.md 规范
   - 验证源码路径和行号范围是否真实匹配
   - 验证整个检索链路是否顺畅无断裂

**判定标准**：
- toc：给定任意 toc.md 中的一条功能描述，应能通过以下路径顺利定位到 spec 和源码：总纲 → 模块索引 → 子模块索引 → spec文档 → 源文件
- spec：从部分描述出发检索到的 spec，是否符合 SPEC-GENERATION-GUIDE.md 规范，功能描述是否详实，核心代码是否真正核心，示例是否可运行，源码行号是否真实匹配
- 如发现描述模糊、路径断裂、指向错误、spec 质量不达标、源码不存在等问题，需立即修正对应文档

每完成一个 toc.md 或 spec 的验证，输出：`[check] {路径} — 通过`

## 关键规则

1. **禁止使用脚本。** 所有分析和生成通过直接读取源文件完成。不得编写或执行辅助脚本。
2. **严格串行处理。** 模块逐个处理（处理完一个才能处理下一个）。模块内子项逐个处理。子模块内spec逐个处理（处理完一个spec才能处理下一个spec）。不得并行化。
3. **综合判断，不要过于依赖目录结构。** 划分模块和子模块时，应综合考虑架构层面（代码组织、职责划分）、业务层面（领域逻辑、功能边界）、目录结构及整体视角。目录结构只是参考之一，关键是理解代码的实际业务和架构内涵。
4. **索引文件 ≤ 500行。** 所有toc.md文件（总纲、模块索引、子模块索引）必须保持精简。它们是索引，不是详细文档。
5. **每个spec必须重新扫描源文件。** 编写spec文档时，必须重新读取实际源文件，不能仅依赖中间规划文件。规划是指导，源码是真理。除了直接相关源文件，还需向上（查看哪些代码调用了这些文件）和向下（查看这些文件依赖了什么）延伸阅读，以充分理解上下文。
6. **核心代码需审慎判定。** 不是所有代码都值得列入spec文档。需判断哪些是真正体现核心逻辑、关键算法、重要接口实现的代码。避免将简单包装、重复、无关键逻辑的胶水代码列入核心代码。
7. **中间文件用于规划。** 所有中间JSON规划文件放在 `{文档树路径}/.tmp/` 下。所有文档生成完成后自动清理 `.tmp/` 目录。
8. **模块/子模块规模。** 一个模块的子项（子模块 + 直接spec）总数 ≤ 30。一个子模块包含 5–40 个spec文档。如果某个领域少于5个spec，应作为模块下的直接spec，而非子模块。
9. **spec文档必须详实专业。** 每个spec文档是对一个或数个相关源文件的完整技术规格说明。必须对每个重要功能/接口/方法进行详细解读，包括功能说明、参数详解、返回值、核心逻辑、核心代码片段。不得敷衍精简。spec文档的质量是整个文档树的核心价值所在。
10. **进度报告。** 完成每个条目时，输出带 `[module]`、`[submodule]`、`[spec]` 前缀的进度信息。
11. **确认后再继续。** 阶段一（模块规划）完成后，向用户展示规划并等待确认，再开始阶段二。
12. **总纲两步走。** 先在阶段四生成初始总纲（模块信息待补充），最后在阶段五-5.6用完整信息更新总纲。
13. **每次生成spec前必须重新读取规范文件。** `references/SPEC-GENERATION-GUIDE.md` 是 spec 文档生成的标准规范。每次处理任何 spec 文档之前都要重新读取该文件，严格遵循其规范。不得依赖记忆。如使用 subagent 处理 spec 任务，**必须在其 prompt 中明确指示 subagent 在生成 spec 之前必须先读取 `references/SPEC-GENERATION-GUIDE.md`**。
14. **全面梳理后再优化总纲。** 所有 spec 生成完成后，必须先全面阅读理解，再从全局视角优化 toc.md。可调整 spec 内容和模块/子模块划分。500行是建议上限，但总纲承担检索器导航职责，信息完整性优先于行数控制，总纲末尾不得添加任何统计信息章节（如"生成统计"、"文档统计"等）。
15. **最终检查索引可追溯性。** 所有文档生成完成后，必须对所有 toc.md 进行索引可追溯性验证：随机抽取各级描述，验证能否顺利定位到 spec 和源码。如有问题立即修正。
