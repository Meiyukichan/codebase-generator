# Spec 文档生成规范

> **重要提醒**：在生成任何 spec 文档之前，**必须重新读取本文件**，严格遵循以下所有规范。本文件是 spec 文档生成的标准参照，不可跳过。

---

## 一、核心原则

1. **必须详实、完整、专业**。spec 文档是对一个或数个相关源代码文件的完整技术规格说明，不是简单的功能列表。
2. **每个 spec 必须重新扫描源文件**。编写 spec 文档时，必须重新读取实际源文件。规划文件只是指导，源码才是真理。
3. **核心代码需审慎判定**。不是所有代码都值得列入 spec 文档。应优先选取真正体现核心逻辑、关键算法、重要接口实现的代码。避免将简单包装代码、重复代码、无关键逻辑的胶水代码列入核心代码。
4. **篇幅应充分，不得敷衍精简**。

---

## 二、源文件阅读范围

生成 spec 文档时，除了阅读该 spec 直接相关的源文件外，还需要向上向下延伸：

- **向上**：查看哪些源文件调用了这些代码，理解该 spec 在外部是如何被使用的
- **向下**：查看该 spec 依赖了哪些其他源文件/模块，理解其底层支撑
- 根据调用关系和使用场景，补充相关上下文到 spec 文档中（如"使用示例"、"依赖关系"等章节）

---

## 三、Spec 文档必须包含的内容

### 3.1 概述

对该 spec 覆盖的业务逻辑/功能的详细描述，包括：
- 该 spec 在系统中的角色和定位
- 设计动机：为什么这样设计，解决什么问题
- 使用场景：在什么情况下会调用到这些功能
- 涉及的源文件列表及其各自职责

### 3.2 整体架构/流程说明（如适用）

如果涉及多步骤流程、状态机、调用链等，需描述整体流程。

### 3.3 功能详情/接口详情

对**每一个**重要功能、接口、方法进行详细解读。每个接口需包含：

- **功能说明**：详细描述，不是一句话。要解释清楚它做了什么、为什么需要它。
- **参数详解**：每个参数的类型、含义、默认值、约束。
- **返回值详解**：类型、含义、可能的错误/边界情况。
- **核心逻辑说明**：这个函数内部做了什么，关键判断和分支。不要只放代码，要有文字解读。
- **核心代码**：从源文件中提取的实际代码片段（15-30行），展示关键实现。
- **代码来源标注**：文件路径 + 行号范围。
- **使用示例**（必须包含）：如何调用该接口、典型输入输出、说明示例代码在做什么。

### 3.4 数据结构

该 spec 涉及的关键类型、接口、枚举、常量，需列出定义和字段说明。

### 3.5 错误处理与边界情况

该 spec 的错误处理策略、异常类型、边界条件。

### 3.6 依赖关系

- **依赖**：该 spec 依赖哪些其他模块/spec
- **被依赖**：哪些其他模块/spec 依赖该 spec

### 3.7 使用示例

展示该 spec 的完整使用场景代码，包括：
- 如何初始化/创建所需对象
- 如何调用核心接口
- 典型的输入输出示例
- 常见的使用模式

并附上逐步说明，读者可以按此直接使用。

## 五、注意事项

1. **spec 文档不是对某个源文件的简单描述**，而是对同一最小业务逻辑、相同逻辑或相关性强的一个或数个源代码文件的完整技术规格。
2. **核心代码片段必须从实际源文件中提取**，不是编造的。
3. **使用示例必须基于实际代码**，不能凭空捏造。
4. **每个接口的使用示例**，是读者理解如何调用该接口的关键。
5. **整体使用示例**，是读者理解如何将该 spec 整合到实际业务中的关键。
